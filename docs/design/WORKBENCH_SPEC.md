## §0 本文的性质

本文档记录 KFM v4 卡片工作台的设计讨论、核心意图和实现方向。

**为什么需要本文**：KFM v4 当前的文件操作链路单一——点击文件树 → 触发命令/事件。没有"收藏"、"暂存"、"批量处理"的概念，也没有多卡片管理与直观切换的机制。本文提出以"购物车模式"为突破口，在文件树与浮卡系统之间建立一条全新的交互链路，同时将已有的卡片堆、浮卡、顶栏、光标等概念整合为统一的工作台体验。

**本文是"活的"**：设计阶段可能会出现新想法、新约束，或者实际实施中发现某个假设不成立。所有发现都应先更新本文，再改代码。本文不追求一次定稿——它应该是随着实现过程逐步收敛的。

## §1 核心设计意图（不可违反）

### 1.1 三条交互链路，同一渲染终点

KFM v4 最终应该有三个入口，都指向同一个卡片渲染系统：

| # | 链路 | 用户操作 | 中间层 | 终态 |
|---|------|---------|--------|------|
| 1 | **快速预览** | 文件树**点击**文件 | —（无中间层） | 预览浮卡（预览略缩态） |
| 2 | **购物车模式** | 文件树**右滑**文件/文件夹 | 右侧购物车抽屉 | 从购物车召唤的浮卡 |
| 3 | **卡片堆唤出** | 右侧卡片堆图标 | —（已有功能） | 编辑器浮卡（详细态） |

三条链路必须共享同一个浮卡系统核心——`launchFileCard()` 或类似函数——但允许不同的初始状态（预览略缩 vs 详细编辑）。

### 1.2 文件浮卡 = 双态架构

文件浮卡不是另起炉灶，而是复用现有的浮卡系统（`floating-card.ts`），**但增加一种新的卡类型和一种新的显示模式**。

**两种状态：**
- **预览略缩态（preview）**：与现有浮卡的 active 态尺寸一致（155×68px），显示文件图标、文件名、简短元信息。点击后不切换状态，**点击 BR 光球才进入详细态**。
- **详细态（detailed）**：展示文件内容或编辑器的完整区域。在不同设备上表现不同：
  - **手机**：全屏覆盖，或下拉的大半屏抽屉
  - **电脑**：大尺寸浮卡，但仍然是浮卡体系的一部分（可拖拽、可缩放）

两种状态之间通过**BR 光球**切换。详细态中有一个按钮（或手势），可以随时回到预览略缩态（即回到桌面视图）。

**未来扩展**：详细态将改造为编辑器，支持不同文件格式（文本、代码、Markdown、图片……）。卡片堆中的"新建文件"卡片也会路由到同款文件编辑器——这意味着编辑器的渲染核心应该是与卡类型无关的独立模块。

### 1.3 购物车模式：去重策略是开放式的

**洛的原话：**
> "滑动一次后，再滑动，就没有效果了。或者更开放一些，滑动一次后，再滑动，就再加一个卡片。方便未来做同文件对比修改，像对同一个文件开了两个窗口。"

初版实现应支持**两种模式**，通过 `KFMState` 中的配置开关控制：

```typescript
interface CartConfig {
  /** 同文件是否允许创建多个实例 */
  allowDuplicates: boolean;  // 默认 false（去重），购物车模式下设为 true
}
```

- **去重模式（allowDuplicates = false）**：文件已在购物车中时，再次右滑无效果（或在 UI 上微反馈——轻闪一下，表示"已经在里面了"）
- **多实例模式（allowDuplicates = true）**：再次右滑时，在购物车中新增一条记录，`instanceIndex` 自增，显示的标签变为"文件名 × 2"、"文件名 × 3"

**未来场景**：多实例模式可直接支持"同文件对比编辑"——同一个文件打开两个编辑器标签页，各自独立修改。

### 1.4 文件夹也是卡片

右滑文件夹时，文件夹整体进入购物车。购物车中的文件夹卡片可以：

- **点击召唤**：以文件夹卡片的形式展示（内部含文件列表预览，或只是一个带文件夹图标的标签）
- **展开查看**：在浮卡中展开文件夹内容
- **未来**：文件夹卡片的详细态可能展示目录树或文件列表

### 1.5 顶栏与光标联动

**洛的原话：**
> "顶端的栏，我想做一个像抽屉一样的，没有内容的时候就不显示，如果有卡片了，不管是卡片堆里的，还是文件树召唤出来的，都可以在上面显示。"
> "点击标签，就像卡片堆的聚焦一样，伸出来一些就够了，然后同时在对应的卡片上加上那个手绘光标，切换的时候同步移动。"

顶栏是一个**状态映射层**——它不拥有卡片，不管理卡片，只反映 `KFMState.openCards` 的状态。它的出现条件是：`openCards.length > 0`。

点击标签的交互效果：
1. 对应浮卡**聚焦**（z-index 提升，行为类似卡片堆的聚焦——伸出来一些位置偏移）
2. 青蓝色手绘光标**移动**到这个浮卡上（被聚焦的上一张卡的光标消失）
3. 顶栏标签同步高亮

这个联动应该由 `KFMState.focusedCardId` 驱动——所有组件（顶栏、光标、浮卡）都订阅这个字段的变化。

## §2 交互全景

以下展示用户从进入文件树到完成操作的全部视觉链路。

```
┌─────────────────────────────────────────────────┐
│  █████████████████████████████████████████████  │  ← 顶栏抽屉（有卡片才出现）
│  [📄 main.ts] [📂 src/] [📄 config.ts × 2]     │     点击标签 → 聚焦 + 光标
└─────────────────────────────────────────────────┘

┌──────────────┐     ┌────────────────────────────┐
│  文件树       │     │         浮卡主屏 / 桌面      │
│              │     │                            │
│  > src/      │     │  ┌─────────┐  ┌─────────┐  │
│    main.ts ──┼──右滑→  │         │  │         │  │
│    utils.ts  │     │  │ main.ts │  │ src/    │  │
│  > docs/     │     │  │ (预览)  │  │ (文件夹)│  │
│    README.md │     │  └────┬────┘  └────┬────┘  │
│              │     │       │             │        │
│              │     │  ┌────▼─────────────▼────┐   │
│              │     │  │     购物车抽屉          │   │
│              │     │  │ [main.ts][src/]         │   │
│              │     │  └────────────────────────┘   │
└──────────────┘     └────────────────────────────┘

用户视角的操作序列（以购物车模式为例）：
1. 文件树中右滑 "main.ts" → 文件飞入右侧购物车（动画同卡片堆召唤反向）
2. 文件树中右滑 "src/" → 文件夹同样飞入购物车
3. 再次右滑 "main.ts"（多实例模式）→ 购物车新增一条 "main.ts × 2"
4. 点击购物车中的 "main.ts" → 卡片从购物车飞出，变为预览浮卡（动画同卡片堆召唤）
5. 预览浮卡出现后，顶栏抽屉出现，显示 "[📄 main.ts] [📂 src/] [📄 main.ts × 2]"
6. 点击 BR 光球 → 切换为详细态（全屏 / 大半屏抽屉 / 大浮卡）
7. 点击顶栏中的 "src/" → 聚焦到文件夹卡片，光标移动到该卡
```

## §3 购物车模式（Cart Drawer）

### 3.1 什么是购物车

购物车是一个**右侧抽屉容器**，与卡片堆共享同一侧的位置和动画系统。它作为"暂存区"，让用户可以：

1. 在文件树中浏览时，把感兴趣的文件/文件夹**滑入**购物车
2. 打开购物车**统一查看**所有暂存的文件
3. 从购物车中点击一个条目，**召唤**对应的浮卡
4. 从购物车中长按一个条目，进入**编辑态**（见 §7）

### 3.2 交互细节

| 操作 | 行为 | 动画 |
|------|------|------|
| 文件树中**右滑**文件 | 文件从文件树位置飞到购物车中 | 同卡片堆召唤浮卡的反向——卡片从文件树"吸入"购物车 |
| 文件树中**右滑**文件夹 | 文件夹整体进入购物车 | 同上 |
| 购物车中**点击**条目 | 条目从购物车飞出，变为预览浮卡 | 同卡片堆召唤浮卡——卡片从购物车"弹出"到桌面 |
| 购物车中**长按**条目 | 条目进入编辑态（高亮、可拖动排序） | 轻微缩放 + 阴影变化 |
| 购物车**空白处点击** | 关闭购物车抽屉 | 抽屉向右滑出 |

### 3.3 核心数据结构

```typescript
interface CartEntry {
  id: string;              // 唯一标识，同文件第二次加入时生成不同 id
  type: 'file' | 'folder';
  path: string;
  name: string;
  instanceIndex: number;   // 第几次加入（用于多实例区分和显示）
  createdAt: number;       // 加入时间（用于排序）
}

interface CartState {
  entries: CartEntry[];
  isOpen: boolean;         // 购物车抽屉是否打开
  editMode: boolean;       // 是否处于编辑态
}
```

### 3.4 与卡片堆的共享

购物车和卡片堆共享：
- 相同的**右侧位置**（两者互斥，要么显示购物车，要么显示卡片堆）
- 相同的**动画系统**（弹性入场、卡片缩放、卡片飞出）
- 相同的**手势注册表**（拖拽、点击、长按）

它们的区别在于：
- **卡片堆**：持久化的卡片集合，内容由用户主动创建/管理
- **购物车**：临时的暂存区，内容来自文件树的右滑操作，关闭即清空（或可配置为持久化）

## §4 文件浮卡（File Card）

### 4.1 复用 vs 新建

文件浮卡**复用**现有的浮卡系统（`floating-card.ts`）。不新建渲染引擎，不改动手势注册表，不改动四角光球逻辑。新增的内容：

1. **一种新的卡模式**：`mode: 'preview' | 'detailed'`
2. **一个创建入口函数**：`launchFileCard(path, name, options?)`

### 4.2 预览略缩态（preview）

| 属性 | 值 |
|------|-----|
| 尺寸 | 与 active 态一致（155 × 68px 左右） |
| 内容 | 文件图标 + 文件名 + 简短元信息（大小/修改日期） |
| 光球 | 四角光球正常显示，BR 光球用于切换到 detailed 态 |
| 交互 | 可拖拽、可缩放、点击内容区域不做状态切换 |

### 4.3 详细态（detailed）

| 属性 | 值 |
|------|-----|
| 手机端 | 全屏覆盖 / 下拉大半屏抽屉 |
| 电脑端 | 大尺寸浮卡（覆盖大部分视口，但仍属于浮卡体系） |
| 切换方式 | BR 光球点击 ←→ 回到预览态的按钮 |
| 内容 | 文件内容预览（文本/代码/图片……） |
| 未来 | 完整的编辑器（文本编辑器、Markdown 渲染、图片查看器……） |

### 4.4 动画

- **预览态 ↔ 详细态**：平滑尺寸过渡动画，从 BR 光球位置展开/收缩
- **手机端全屏**：从浮卡位置向上/向下展开（类似 modal sheet）

## §5 顶栏抽屉（Top Bar）

### 5.1 定位与行为

- **位置**：`position: fixed; top: 0; left: 0; width: 100%`
- **出现条件**：`KFMState.openCards.length > 0`
- **隐藏条件**：`KFMState.openCards.length === 0`（零开销，不渲染）
- **高度**：默认 48px（可根据内容自适应）
- **层级**：始终在所有内容之上（z-index 最高）

### 5.2 内容

每个 `openCard` 对应一个标签（Tab）：

```
[📄 main.ts] [📂 src/] [📄 config.ts × 2] [+]
```

- **文件卡片**：`📄 文件名`
- **文件夹卡片**：`📂 文件夹名`
- **多实例**：`📄 文件名 × N`
- **右侧 `+` 按钮**（可选）：点击唤起更多操作（新建卡片、清空购物车……）

### 5.3 交互

| 操作 | 行为 |
|------|------|
| 点击标签 | 对应浮卡聚焦，顶栏标签高亮，手绘光标移动到该浮卡 |
| 左右滑动顶栏 | 滚动标签列表（当卡片过多时） |
| 长按标签 | 卡片预览（类似卡片堆的 peek） |
| 标签右键/双指点击 | 关闭该卡片（从 KFMState.openCards 移除） |

### 5.4 状态驱动

顶栏不直接管理卡片。它的状态完全来自 `KFMState` 的订阅：

```typescript
KFMState.subscribe(state => {
  if (state.openCards.length === 0) {
    topBar.hide();
  } else {
    topBar.render(state.openCards);
    topBar.highlightCard(state.focusedCardId);
    topBar.show();
  }
});
```

## §6 手绘蜡笔光标（Hand-Drawn Cursor）

### 6.1 设计理念

**洛的原话：**
> "光标，我想有一个设计，就是用手绘风格，青蓝色的，像小孩子用蜡笔在上面画了个圈。最好是可以随机化的。"

这个设计传递的气质：
- **亲和感**：手工感 vs 冰冷技术感
- **强调而非边框**：不是"选中框"，而是"被人圈出来了"
- **艺术感**：与项目现有的科技冷色调形成温暖对比

### 6.2 技术实现方案

使用 **SVG overlay** 绘制在浮卡之上：

```typescript
interface DoodleCropOptions {
  box: {
    x: number;
    y: number;
    width: number;
    height: number;
  };
  seed: number;  // 随机种子，同一个卡片聚焦时保持稳定
}

function generateDoodleCropPath(options: DoodleCropOptions): string {
  const { box, seed } = options;
  const rng = new SeededRandom(seed);
  
  // 1. 在 box 的四条边上生成 8~12 个采样点
  // 2. 为每个采样点施加随机偏移（6~16px）
  // 3. 用二次贝塞尔曲线连接各点
  // 4. 在首尾之间预留 3~10px 缺口（蜡笔感）
  
  return svgPath;  // SVG path d 属性
}
```

**随机维度（可控）：**

| 维度 | 范围 | 影响 |
|------|------|------|
| 采样点数量 | 8 ~ 12 个 | 圈的光滑 vs 粗糙程度 |
| 偏移幅度 | 6 ~ 16px | 圈的"抖动"程度 |
| 线条透明度 | 0.4 ~ 0.8 | 蜡笔的"用力不均匀"感 |
| 线条宽度 | 2 ~ 4px | 蜡笔的粗细 |
| 颜色（HSL 色相） | 180° ~ 200° | 青蓝色范围的深浅波动 |
| 首尾缺口 | 3 ~ 10px | 圈未闭合的"手绘感" |

**性能策略：** 聚焦切换时生成一次 SVG，缓存，不随帧动画重绘。

### 6.3 额外效果（未来可扩展）

- **移动时的拖尾轨迹**：光标从一张卡移动到另一张卡时，画一道淡出的轨迹线（类似于蜡笔在纸上拖动后留下的痕迹）
- **双重圈**：有时候画出两个圈后又划掉一个的不确定感
- **圈内微表情**：在圈内的右上角画一个小小的星星或爱心（可选，随机出现）

## §7 文件树编辑态（KFMv2/v3 参考）

### 7.1 参考来源

洛提到 KFMv2/v3（位于 `/root/kfm/`）中曾做过文件树的编辑效果。那部分代码应该包含：

- 文件/文件夹的拖拽排序
- 长按进入多选模式
- 批量操作（移动、复制、删除）
- 右键/手势菜单

### 7.2 购物车编辑态

购物车模式和文件树编辑态的结合点：
- 购物车中**长按**条目 → 进入编辑态
- 编辑态中可**拖动排序**购物车中的条目（决定后续召唤浮卡时的 z-index 顺序）
- 编辑态中可**批量操作**（全部召唤、全部清除、选中后移动……）

### 7.3 文件树本身的编辑

购物车模式本身不修改文件树的渲染。文件树的编辑效果（拖拽排序、重命名、右键菜单等）应该作为**独立的后续功能**，在购物车模式稳定后再实现。

初版购物车模式中，文件树的唯一变化是：**右滑手势被注册用于触发购物车操作**。

## §8 状态模型（KFMState 扩展）

```typescript
// ========== openCards（顶栏 + 浮卡系统） ==========

interface OpenCard {
  id: string;
  type: 'file' | 'folder' | 'editor';
  path: string;
  name: string;
  instanceIndex: number;      // 同文件多实例用（default 0）
  mode: 'preview' | 'detailed';
  createdAt: number;
}

// ========== cart（购物车） ==========

interface CartEntry {
  id: string;
  type: 'file' | 'folder';
  path: string;
  name: string;
  instanceIndex: number;
  createdAt: number;
}

interface CartState {
  entries: CartEntry[];
  isOpen: boolean;
  editMode: boolean;          // 购物车编辑态
}

// ========== KFMState 扩展 ==========

interface KFMState {
  // ... 现有字段

  // 新增：卡片管理器
  openCards: OpenCard[];
  focusedCardId: string | null;

  // 新增：购物车
  cart: CartState;

  // 新增：配置
```

## §10 开放问题与待确认项

以下问题在初版设计阶段尚未最终确定，标记为 [OPEN] 供后续讨论。每个问题在解决后应更新本文档。

### 10.1 购物车持久化

- [OPEN-001] **购物车条目是否应该持久化？**
  - 选项 A：关闭购物车即清空（纯粹临时暂存）
  - 选项 B：持久化到 localStorage，下次打开浏览器会话时恢复
  - 当前偏向：选项 A（简单、可预期）

### 10.2 购物车 vs 卡片堆的关系

- [OPEN-002] **购物车和卡片堆是否应该合并？**
  - 选项 A：保持分离——购物车是临时暂存区，卡片堆是持久收藏区
  - 选项 B：统一为一个概念——"我的卡片"包含所有入口（文件树 + 卡片堆）
  - 当前偏向：选项 A（分离，职责清晰）

### 10.3 文件树右滑手势冲突

- [OPEN-003] **文件树中的右滑是否与页面整体的右滑返回冲突？**
  - 解决方案 A：仅当文件树处于激活状态时，右滑才触发的购物车操作
  - 解决方案 B：在文件树区域内捕获右滑事件，不冒泡到页面级别
  - 需要实际测试后决定

### 10.4 多实例上限

- [OPEN-004] **同文件最多可以创建多少个实例？**
  - 选项 A：不限（用户可以开 N 个窗口）
  - 选项 B：限量（比如最多 5 个）
  - 当前偏向：不限，但在顶栏中通过 "× N" 明确指示，超过 5 个时 UI 加入滚动

### 10.5 手绘光标性能

- [OPEN-005] **手绘光标的 SVG 渲染是否应该在低端设备上降级？**
  - 选项 A：不做降级，SVG 本身就是轻量的
  - 选项 B：检测设备性能，弱设备上使用纯 CSS 边框替代
  - 当前偏向：选项 A（先做完整效果，如果出现性能问题再考虑降级）

### 10.6 大页面编辑器

- [OPEN-006] **编辑器的具体文件格式支持范围？**
  - 初版建议：文本预览 → 纯文本 + Markdown（只读）
  - 未来：代码编辑器（Monaco/CodeMirror）+ Markdown 渲染 + 图片查看器 + JSON 查看器
  - 需要在 Phase 4 正式开始前讨论

### 10.7 与 AI 操作的对称性

- [OPEN-007] **AI 如何通过内部眼睛操作这些新功能？**
  - 例如：AI 发送 `cart-add /path`、`cart-clear` 等命令
  - 建议在 Phase 1 完成后，作为 BOX_LOCATION_MAP_SPEC.md 的延伸功能加入

### 10.8 顶栏溢出

- [OPEN-008] **当卡片过多顶栏放不下时？**
  - 选项 A：左右滑动标签列表（类似浏览器的 tab scroll）
  - 选项 B：多行显示（顶栏变高）
  - 选项 C：收缩为下拉菜单（"查看更多"）
  - 当前偏向：选项 A（最直觉，与移动端交互一致）
