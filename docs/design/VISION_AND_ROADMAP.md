## 一、核心理念

### 1.1 本质

**AI 人机交互的个人工作台，创意与能力的延伸工具。**

KFM 不是一个"文件管理器"，也不是一个"AI 聊天客户端"。它是你在手机上的私人工作台——文件浏览、AI 对话、代码编辑、终端操作，都以"卡片"的形式在工作台上浮动，随时展开收起、随手调配。

### 1.2 一切皆卡片

所有浮动元素都是卡片。AI 对话、调试面板、文件编辑器、工具面板——**唯一的区别是内容**。同一个交互框架支撑所有场景：

- 相同的拖拽行为
- 相同的展开/收起状态机（`float` ↔ `dock` ↔ `editing`）
- 相同的标题栏 + 内容区结构
- 相同的毛玻璃 + 渐变边框 + 四角光球视觉

### 1.3 可插拔的 Agent 引擎（核心架构决策）

KFM **不做 Agent 框架**。它不自己实现 LLM function calling、不写工具路由、不管理 Agent 会话上下文。这些职责外包给成熟的后端 CLI 工具。

```
┌───────────────────────────────────────────────┐
│          KFM 卡片工作台（你的 UI）                │
│                                                │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐          │
│  │对话卡│ │文件卡│ │调试卡│ │设置卡│          │
│  └──┬───┘ └──────┘ └──────┘ └──────┘          │
│     │                                          │
│  ┌──▼──────────────────── Agent 适配层 ──────┐ │
│  │  OMP  │  Claude Code  │  Deepseek  │  …    │ │
│  │  (子进程，每任务可 spawn 多个，并行)        │ │
│  └────────────────────────────────────────────┘ │
│                                                │
│         KFM 服务端（文件 API + exec + grep）      │
└───────────────────────────────────────────────┘
```

| 角色 | 名称 | 职责 |
|------|------|------|
| **管家** | KFM Shell（主光球面板） | 环境管理、Agent 编排、配置持久化、密钥管理、工作区记忆 |
| **助手** | OMP / Claude Code 等 | 执行具体任务：读代码、写代码、构建验证、搜文件 |
| **外壳** | KFM 卡片 UI | 浮卡、毛玻璃、手势、动画——给 Agent 穿的外衣 |

关键约束：
- KFM 自己不实现任何 LLM 协议。它 spawn 子进程，把用户输入转发过去，把输出流式渲染回来。
- Agent 之间不需要知道彼此存在。它们通过**同一份文件系统**共享上下文。
- 用户可以同时开多个助手（一个写代码、一个查文档），各自独立进程，互不阻塞。

### 1.4 用户和 AI 对称操作

深层设计逻辑：**用户能做什么，AI 就能做什么**。

- 用户手指拖拽卡片 → 直接操作 UI
- AI 通过适配层调用卡片 → 触发相同的行为

能力不是预注册的"AI 工具集"，而是跟着卡片走的。创建一张新卡片的同时，用户和 AI 都获得了新能力。每张卡片就是 AI 的一个工具，不需要单独注册。

### 1.5 "没有主页面"——只有当前卡片

当前布局不是妥协，而是为卡片化做的空间预留：

```
底部:   全局输入栏（常驻入口，随时呼叫 AI）
中央:   工作台网格（放卡片的空间）
侧栏:   收起（需要时唤出 = 文件树卡片展开）
光球:   收起（需要时展开 = AI 对话卡片）
```

不存在"主页面"的概念，只有"当前操作的卡片"——打开侧栏等于展开了文件树卡片，展开光球等于展开了对话卡片，收起全部等于所有卡片最小化，留出空白工作台。

### 1.6 标记系统（为手机屏幕设计）

手机屏幕小，用户不能像桌面端那样"展示"多个窗口给 AI 看。所以用"标记"来告诉 AI 上下文——反而比桌面更精确，因为必须显式标记：

| 模式 | 数量 | 语义 | 实现状态 |
|------|------|------|---------|
| 点击光标 | 1 个 | "操作目标在这里" | 文件树中已实现；扩展到卡片系统待实现 |
| 路标标记 | 多个 | "参考这些内容" | 待实现 |

### 1.7 AI 自我驱动的自指循环

这是一个重要的设计洞察：**KFM 的代码由 AI agent 反复迭代写出。卡片系统 UI 完成后，第一个用它来自我迭代的项目就是 KFM 本身。**

也就是说：KFM 的代码由 agent 写 → agent 跑在 KFM 卡片里 → agent 通过 KFM 的工具继续改 KFM 的代码。项目是一个**自指系统**。你设计的 UI 即是这个循环的镜子。

这意味着：
- CardManager + 插件系统一旦完成，AI 应该能直接通过它来创建新卡片、注册到工具坞、验证构建
- KFM 本身的改进可以由 KFM 自己完成
- 你的设计文档和规范（本章）是 AI 理解如何操作这个循环的唯一起点

## 二、整体架构

### 2.1 四层模型

```
┌──────────────────────────────────────────────────┐
│                 ① 卡片工作台 UI                     │
│   卡片系统（shell/dock/manager）+ 浮卡 + 输入栏     │
│   手势系统、动画渲染、光球视觉                      │
├──────────────────────────────────────────────────┤
│                 ② Agent 适配层                     │
│   配置管理（API key / 模型选择 / Agent 类型）        │
│   子进程管理（spawn / stdin / stdout / 生命周期）     │
│   消息格式转换（输入 → Agent 格式 / 输出 → 卡片渲染）  │
├──────────────────────────────────────────────────┤
│                 ③ Agent 引擎                      │
│   OMP / Claude Code / Deepseek TUI / 其他         │
│   承载 LLM function calling、工具调用、循环推理     │
├──────────────────────────────────────────────────┤
│                 ④ KFM 服务端                       │
│   Express + 文件 API + exec 端点 + grep 端点      │
│   nginx 反代（127.0.0.1:8021）                    │
└──────────────────────────────────────────────────┘
```

### 2.2 依赖方向

```
UI 层 → 适配层 → 服务端/Agent
```

UI 层不直接调用 LLM API。它只跟适配层对话。适配层决定把请求发给哪个 Agent，然后把返回的内容传回 UI 层渲染。

Agent 之间的通信通过文件系统隐式完成：
- Agent A 写了一个文件 → Agent B 可以读到
- Agent A 改了代码 → npm run build 的结果两个 agent 都能看到
- 不需要跨 agent 直接通信协议

### 2.3 当前技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 语言 | TypeScript 6 + ES2022 | 全栈 ESM |
| Canvas 渲染 | 自研引擎（Box → Renderer） | v2 引擎，不依赖第三方 Canvas 框架 |
| 动画 | GSAP 3.15 | 通过 `animation-registry.ts` 隔离，禁止直接 import gsap |
| 文本测量 | `@chenglou/pretext` | 零 reflow 离屏测量与换行 |
| 布局 | 自研 flex.ts | 轻量 Flexbox 计算 |
| 边框绘制 | BorderDrawer.ts | 四边独立控制 + 宽度渐变 |
| 服务端 | Express 4 | 文件读写 API，走 nginx 反代到 `127.0.0.1` |
| 构建 | esbuild 0.28 + TypeScript 6 | `tsc --noEmit` 类型检查 → esbuild 打包 |

## 三、交互模型

### 3.1 全局输入栏

输入栏是**唯一的入口**，卡片系统是**输出渲染器**。不区分"命令面板"和"聊天框"——同一个输入框，按 Tab 进入命令模式，按 Enter 发送对话：

```
正常打字 → 回车 → 发送给当前 Agent
输入 /grep "xxx" → 直接返回搜索结果（命令模式）
输入 @代码助手 xxx → 指定 Agent 处理
输入 /card 设置 → 展开设置卡
```

这个交互逻辑需要和 Agent 适配层配合实现。

### 3.2 卡片生命周期与状态

```
  工具坞 ---拖出-->  浮动卡片  <---左栏点击（文件卡）
     ^                  |
     |                  | 点击标题栏（或 BR 光球）
     |                  v
     +--拖回/关闭--- 展开面板
                       |
                       | 长按 BR 光球（600ms）
                       v
                    编辑大小（缩放模式）
                       |
                       | 松手
                       v
                    退出编辑
```

| 状态 | 说明 | 视觉 |
|------|------|------|
| `dock` | 停靠在右侧工具坞 | 仅显示彩色圆点 + 图标 |
| `float`（compact） | 紧凑态，仅右下角光球 | 120×36 |
| `float`（active） | 展开态，四角光球齐全 | 155×68，内容可见 |
| `editing` | 长按进入调整大小 | 加亮阴影 + 缩放手柄 |

### 3.3 卡片堆的定位

卡片堆不是工具坞——它是**快速选择器**。从右边缘左滑唤出，上下滑切换聚焦，左滑发射为浮动卡片：

```
右边缘 30px 内水平左滑    → 呼出堆叠卡片面板
卡片区域垂直滑动          → 切换聚焦卡片
聚焦卡片左滑              → 飞入工作台（成为浮动卡片，compact 态）
右滑（或点击背景）        → 关闭堆叠面板
```

手势判定区分离：
- 边缘区（屏幕右沿 30px 内）→ 只判水平滑动（呼出/关闭）
- 内区（卡片区域）→ 只判垂直滑动（切换选择）
- 两个区域不混淆

卡片堆中的卡片和浮动卡片是**同一个视觉原型的不同状态**。卡堆中的预设卡片未来应对接插件系统，动态填充内容。

### 3.4 光球 = Agent 指示灯

每个 Agent 或卡片可以关联一个光球。光球颜色标识对应的服务：

| 光球 | 颜色 | 对应 |
|------|------|------|
| 主光球 | 紫色 | KFM 管家（主入口） |
| 浮卡 BR 光球 | 随卡片配色 | 对应卡片 |

卡片收起时只露光球，用户看到颜色就知道哪个 Agent/卡片在工作。

### 3.5 消息渲染器

对话卡的核心内容区是一个**消息流渲染器**。它不关心消息来自谁——用户发送的、Agent 返回的、工具执行结果的——都渲染成不同的气泡样式。

```
渲染器规范（待定）：
  ┌─ 用户消息 ─────────────────────┐
  │  靠右，用户色，文本+附件          │
  └─────────────────────────────────┘
  ┌─ Agent 回复 ────────────────────┐
  │  靠左，Agent 色，markdown+代码块  │
  └─────────────────────────────────┘
  ┌─ 工具调用记录 ──────────────────┐
  │  灰底小字: "调用了 read('a.ts')" │
  └─────────────────────────────────┘
  ┌─ 工具执行结果 ──────────────────┐
  │  带折叠的代码块/文件diff/截图    │
  └─────────────────────────────────┘
```

这个渲染器是通用的——任何卡片如果有流式输出需求，都可以复用。

## 四、卡片插件系统

### 4.1 插件是什么

一张卡片 = 一个 `CardDefinition` + 内容插槽的实现。插件就是一个 `.ts` 文件，导出 `CardDefinition` 对象。

卡片目录在项目根 `cards/plugins/` 下（不在 `src/client/` 内，受 tsconfig rootDir 限制）：

```
cards/plugins/
├── debug-card/
│   └── index.ts
└── ...  // 加新插件 → 新建一个目录
```

### 4.2 CardDefinition 接口（待定稿）

```typescript
interface CardDefinition {
  // 标识
  id: string;              // 唯一 ID，如 "ai-chat"
  name: string;            // 显示名称，如 "AI 对话"
  icon: string;            // 标题栏图标（Emoji 或 SVG）
  accentColor: string;     // 主色 → 边框渐变 + 光球

  // 几何
  defaultWidth: number;
  defaultHeight: number;
  minWidth?: number;
  minHeight?: number;

  // 初始状态
  initialState?: 'dock' | 'compact' | 'active';

  // 核心：内容渲染
  render(shell: CardShell): void;
  // shell 提供：
  //   shell.contentEl — HTMLElement，卡片内容插槽
  //   shell.headerEl  — HTMLElement，标题栏（可自定义）
  //   shell.cardEl    — HTMLElement，完整卡片容器
  //   shell.setState('compact' | 'active' | 'editing')
  //   shell.resizeTo(w, h)

  // 生命周期
  destroy?(): void;        // 卡片被移除时清理事件监听
  onResize?(w: number, h: number): void;  // 用户拖拽缩放时回调

  // Agent 集成（可选）
  actions?: ToolDef[];     // 卡片支持哪些操作 → Agent 可见

  // 持久化（可选）
  persistGeometry?: boolean;
}
```

### 4.3 插件注册方式（构建时自动扫描）

由于项目使用 esbuild 打包为单文件 `bundle.js`，无法动态加载文件。插件注册通过**构建时自动扫描**实现：

```
project root/
├── cards/                          ← 公共插件目录（社区下载放这里）
│   ├── ai-chat.card.ts
│   ├── terminal.card.ts
│   └── README.md                   ← "把 .card.ts 丢进来，npm run build 即可"
├── src/client/cards/
│   ├── registry.ts                 ← 手动注册内置插件
│   └── plugins/                    ← 内置核心插件
```

`build.mjs` 中扫描 `cards/` 目录，自动生成注册代码：

```javascript
// 伪代码 — build.mjs 中
const files = readdirSync('cards/').filter(f => f.endsWith('.card.ts'));
const imports = files.map((f, i) =>
  `import * as card${i} from '../../cards/${f}';`
).join('\n');
const registrations = files.map((_, i) =>
  `  card${i}.default,`
).join('\n');

writeFileSync('src/client/cards/_generated.ts', `...`);
```

生成的 `_generated.ts` 被 `.gitignore` 忽略。打开效果：

```
下载 terminal.card.ts → 丢进 cards/ → npm run build → 刷新浏览器 → 卡片已在卡片堆中
```

不需要手动改任何代码。一行 import 都不用写。

## 五、Agent 适配层

### 5.1 适配层职责

Agent 适配层是 KFM UI 和第三方 Agent 引擎之间的薄胶水层。它的职责：

```
用户输入（来自对话卡）
    ↓
适配层：
  1. 查找当前配置的 Agent（OMP / Claude Code / 自定义）
  2. 将用户输入转为该 Agent 接受的输入格式
  3. spawn 子进程（或复用已有会话）
  4. 将 Agent 的 stdout 流式转发给渲染器
  5. 如果 Agent 需要调工具（read/write/exec），由 Agent 自己完成
     （不经过 KFM 的路由）
    ↓
渲染器将响应渲染为对话气泡
```

注意第 5 点——Agent 的工具调用由 Agent 自己管理。KFM 适配层不拦截、不路由工具调用。Agent 直接调文件 API 或 shell 端点。适配层只关心**输入和输出流的格式转换**。

### 5.2 Agent 配置

通过设置卡管理。每个 Agent 配置包含：

```typescript
interface AgentConfig {
  name: string;              // "代码助手"
  type: 'omp' | 'claude' | 'custom';
  command: string;           // 启动命令，如 "omp"
  model?: string;            // 模型选择，如 "deepseek-chat"
  apiKey?: string;           // 存 localStorage（加密？）
  cwd: string;               // 工作目录
  permissions: ('read' | 'write' | 'exec')[];

  // 自定义 Agent 的协议适配
  inputFormat?: 'stdin' | 'http';
  outputFormat?: 'stream' | 'json';
}
```

### 5.3 多 Agent 并行

用户可以同时开多个 Agent，每个在对话卡里有一个独立"标签页"或"子对话"：

```
┌─ KFM 管家（主对话）────────────────────────┐
│  你: 帮我查一下这个 bug                       │
│  管家: 好，我让代码助手查                    │
│  ┌─ 代码助手 (OMP) ──────────────────────┐  │
│  │  (后台，用户看得到实时流式输出)          │  │
│  └───────────────────────────────────────┘  │
│  你: 同时帮我整理一下这个 md                  │
│  管家: 好，让文档助手处理                    │
│  ┌─ 文档助手 (Claude) ───────────────────┐  │
│  │  (另一个进程，并行)                     │  │
│  └───────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

### 5.4 适配层 vs. 自建 Agent 框架

| | KFM 适配层 | 自己造 Agent 框架 |
|--|------------|-------------------|
| 代码量 | ~50 行（输入输出格式转换） | 数百行（tool call 路由 + 上下文管理 + 重试机制） |
| 维护成本 | 零（变更只影响配置） | 持续维护 |
| 灵活性 | 只能选现有 Agent | 可以任意定制 |
| 优势 | 成熟 Agent 已经踩过所有坑 | 控制权在手 |

**结论**：用适配层。Agent 引擎交给成熟方案。

### 5.5 AI 可调用指针

AI 需要一种方式"指给用户看"——不依赖卡片聚焦或状态联动，
只是一个层面最高的独立视觉元素。

**设计：**
- 层级：`position: fixed; z-index: 9999; pointer-events: none`——不阻挡任何交互
- 移动：GSAP `to(pointer, { x, y, duration: 0.35, ease: 'power3.out' })` 平滑过渡
- 外观：待定（手指 SVG / 圆点+涟漪 / 手绘圈 / pin），可后续切换

**AI 命令协议：**

| 命令 | 参数 | 说明 |
|------|------|------|
| `point-to` | `x y` 或 `selector` | 指针移动到屏幕坐标或匹配元素中心 |
| `hide-pointer` | — | 隐藏指针 |
| `show-pointer` | — | 显示指针 |

**依赖**：AI 工具管线就位后（ws-channel 命令通道 + 页面布局感知）方可实施。

### 6.2 配色系统

沿用卡片堆的双色独立随机机制。每种卡片类型分配 Accent Color，渐变方向固定 135deg（右上→左下）：

| 卡片 | 色值 | 用途 |
|------|------|------|
| AI 对话 | `#7c3aed` 紫 | 全局常驻 |
| 调试面板 | `#00c8a0` 青 | 工具坞 |
| 文件编辑 | `#50A880` 绿 | 文件类型映射 |
| 设置 | `#C88C5A` 琥珀 | 工具坞 |
| 新建文件 | `#5088C8` 蓝 | 工具坞 |
| 笔记 | `#B4AA50` 金 | 工具坞 |
| 插件 | `#9650C8` 紫罗兰 | 工具坞 |
| 统计 | `#6C5CC8` 靛蓝 | 工具坞 |
| 关于 | `#B46478` 玫瑰 | 工具坞 |

- color1（渐变起点）对应：右侧光球（TR、BR）、图标背景
- color2（渐变终点）对应：左侧光球（TL、BL）、图标数字
- 文字颜色统一白色，不派生自卡片色

### 6.3 双层 DOM 结构

```html
<div class="card-shell" style="left:X; top:Y; width:W; height:H">
  <!-- 外层壳：渐变边框（通过 padding 挤出） -->
  <div class="card-border-layer" style="padding: 1px; border-radius: 12px;
       background: linear-gradient(135deg, color1, color2);">
    <!-- 内层：毛玻璃内容区（flow 布局，不绝对定位） -->
    <div class="card-inner" style="border-radius: 11px;
         background: rgba(20,16,32,0.92);
         backdrop-filter: blur(16px);">
      <div class="card-header" style="border-left: 3px solid {accent}">
        <span class="card-icon">{icon}</span>
        <span class="card-name">{name}</span>
      </div>
      <div class="card-content"><!-- 内容插槽，由插件填充 --></div>
    </div>
  </div>
</div>
```

**必须遵守的约束**：
- 边框和背景**必须正交解耦**：外层壳负责渐变边框，内层元素负责毛玻璃。修改背景透明度不丢失边框，修改渐变色不干扰背景。
- 内层毛玻璃必须用 `flow` 布局（由外层壳的 `padding` 自然约束），不能用 `inset:0` 的 absolute 定位。
- 浮卡和卡片堆卡片的 DOM 结构必须一致。

## 七、关键设计决策（历史记录）

以下决策是通过多轮试错和讨论沉淀下来的。后面的人不要轻易推翻它们。

### 7.1 为什么 KFM 不自建 Agent 框架

**背景**：项目最初有"把 OMP 逻辑移植进来"的想法——在 KFM 内实现 tool call 路由、LLM function calling、Agent 循环。

**决定**：不做。KFM 通过 spawn 子进程的方式调用成熟的 CLI Agent（OMP / Claude Code / 其他），只做 UI 壳 + 输入输出转换。

**原因**：
- 不同 Agent 对同一任务的理解和执行能力差异很大（经过多次实测验证）
- 自己写 Agent 框架 = 重新踩一遍别人已经踩过的坑（上下文管理、错误恢复、模型适配）
- Agent 框架的适配成本低（50 行左右的输入输出格式转换），而重新实现的成本高（数百行+持续维护）
- 可插拔架构允许用户根据任务选择最趁手的 Agent

### 7.2 为什么卡片内容插槽用 DOM 而不是 Canvas

**决定**：卡片内容区用 DOM（不是 Canvas 渲染）。

**原因**：
- Canvas 适合渲染"固定排版"的文本和形状，不适合交互式表单、输入框、滚动列表
- 缩放时 DOM + flex 自动重排，Canvas 需要手动计算位置
- AI 对话消息、日志面板、文件编辑器这些核心卡片内容都是交互密集型，DOM 是自然选择

**例外**：文件树仍在 Canvas 中渲染（已有自研引擎），不迁移到卡片系统。

### 7.3 为什么不用动态 import 加载插件

**决定**：插件通过构建时自动扫描注册，而不是运行时动态加载。

**原因**：
- esbuild 打包为单文件 iife bundle，`import()` 动态路径不被支持
- 动态加载需要 webpack chunk splitting 或 esbuild 的 code splitting 模式，会增加构建复杂度和运行时延迟
- 构建时扫描的方案（`cards/` → 自动生成注册代码）零运行时开销，且不需要用户手动改代码

### 7.4 为什么卡片堆的卡片从预设改为插件驱动

**决定**：卡片堆中当前 7 张预设卡（AI 对话、调试面板、文件编辑等）未来将改为插件系统填充。

**原因**：
- 预设卡是"空壳"——只有名字、图标、颜色，没有实际功能
- 真正的功能代码（聊天、日志）当前在 `orb.ts` 和卡片堆日志卡中，需要迁移
- 插件化后，第三方开发者（或 AI 自己）写好一张新卡就自动出现在卡片堆中

## 九、未解决的问题

### 9.1 CardShell 的能力边界

CardShell 给插件暴露哪些能力？

```
已知的：
  - shell.contentEl / shell.headerEl / shell.cardEl
  - shell.setState('compact' | 'active' | 'editing')

待定：

  - 插件能在自己的 corner orb 上注册点击事件吗？
  - 插件能通过 shell 拿到当前主题（theme.ts）吗？
  - 插件能注入自定义 CSS 吗？还是只能复用 card-shell 的样式？
```

### 9.2 插件样式管理

插件开发者如何写样式？

```
方案 A：插件自带 CSS → 在 render() 中创建 <style> 注入
  优点：灵活，插件完全自包含
  缺点：样式冲突风险，卸载时需要清理

方案 B：卡片系统提供固定模板样式 → 插件只填充内容 DOM
  优点：视觉统一，无冲突风险
  缺点：限制了插件的视觉表达

当前倾向：方案 B，但插件可以通过 shell 获取 class name 前缀来做细粒度定制。
```

### 9.3 Agent 适配层的输出格式

不同 Agent 的输出格式不同（纯文本 / markdown / json / 流式 chunks）。适配层是否需要统一为一种内部格式再交给渲染器？还是让渲染器支持多种输入格式？

```
倾向：适配层负责转换 → 渲染器只接受统一格式。
但"统一格式"具体是什么（自定义消息协议？JSONL？）未定。
```

### 9.4 权限管理

插件（尤其是社区下载的）需要访问哪些资源？谁审批？

```
资源类型：
  - 文件系统（读/写）
  - shell 执行
  - git 操作
  - 网络请求
  - 其他卡片的数据

审批方式：
  - 首次使用时弹窗确认 → 记住选择
  - 在设置卡里显式配置
  - 不限制（个人工具，信任本地插件）
```

### 9.5 AI 对话卡的隐藏和常驻

AI 对话卡是始终可见（即使收起也能快速唤出），还是可以被用户关掉？

```
问题：如果支持关闭，用户如何重新打开？
  - 工具坞里有 AI 图标 → 点击重新唤出
  - 全局输入栏常驻 → 输入任何内容自动唤出

问题：是否允许用户同时打开多个 AI 对话卡？
  - 可以（多 Agent 并行，每个一个标签页）
```

### 9.6 卡片位置持久化

卡片的位置、大小是否跨会话保持？

```
存储位置：
  - localStorage（浏览器本地，简单）

哪些需要持久化：
  - 每张卡片的 left / top / width / height
```

```
更新：替换 .card.ts 文件 → npm run build → 卡片刷新

问题：升级后旧的 localStorage 数据结构不兼容怎么办？
```

### 9.8 会话连续性

Agent 会话到期后，新 Agent 如何"接上"当前进度？

```
已有机制：
```
