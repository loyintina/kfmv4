# KFM v4 — AI 修改约束协议（Harness）

> **在修改任何 .ts 文件前，AI 必须逐条通读本文档并内化所有约束。**
> 未读本文档就动手的修改，大概率方向错误。
> 本文件是项目的不变量（invariants）和修改协议，不是参考手册。

---

## 〇、心法原则（思考框架）

以下 7 条原则是执行所有改动前必须内化的思考方式。违反任何一条的修改，大概率需要重做。

### 1. 先问"为什么"，再问"改哪里"
不要急于修复症状。找到根因，一个根源问题往往能解释所有表面症状。用补丁盖住问题，只会让问题以另一种形式在其他地方出现。

### 2. 从源头上简化，不是在输出端增加
数据模型只存不可继续简化的原始信息。所有派生状态在消费时实时计算，而不是预先存储。一个好的重构应该让代码变少，而不是变多。

### 3. 不要跨模块依赖
一个模块的决策边界等于它的可见边界。如果 A 需要知道 B 的状态才能做决策，说明设计有问题。每个模块都应该是独立的颜色单位。

### 4. 状态在展示前就位
不要让用户看到"变化的过程"。过渡动画服务于状态切换，但状态本身应该在动画开始前已就位。

### 5. 代码越改越少是正向指标
把 30 个变量变成 3 个，删除 10 个函数留下 5 个——这才是优雅。每次修改都应该让系统变得更简单。如果改完后代码变多了，大概率方向错了。

### 6. 先理解整体，再拆解局部
不要一头扎进代码细节。先看全貌，理解架构脉络，再定位要改的地方。理解 → 诊断 → 方案 → 执行，顺序不要乱。

### 7. 选择最匹配的一个方案，不列清单
分析场景，权衡取舍，给出你推荐的那一个，附上理由。如果有备选再说一句"如果想走另一条路也可以"。

## 一、项目核心约束（违反就是 bug）

### 1.1 动画安全

```
▎ 动画开始前相关状态应处于初始态（tree-render: _activeOverlays === []）
▎ 动画结束后相关状态应回到初始态
▎ rebuildTree() 被调用时 L.isAnimating 应为 false
    例外：超过 3000ms 超时兜底，允许强制释放
▎ 每轮动画结束时必须调用 _resetAnimTimeline()
    ts.clear() + ts.time(0) + ts.reversed(false)
```

### 1.2 类型安全

```
▎ Overlay 元数据用 (as Box & OverlayMeta) 访问，禁止 (as any)._xxx
▎ 已知遗留: _createVisualClone 中 (src as any).data 一处逃逸 (P2)
▎ 所有 _ 前缀属性只在模块内访问
    跨模块读写 _ 前缀属性应通过显式接口
▎ FloatingCardItem 的 orb 字段类型为 HTMLElement | null
    紧凑态下 tl/tr/bl 确实为 null，访问前必须空值守卫
```

### 1.3 依赖方向

```
renderer-lifecycle.ts (L)
       ↓
canvas-utils.ts         ← 只依赖 L
       ↓
canvas-cursor.ts        ← 不导入 tree-*
       ↓
canvas-scroll.ts        ← 不导入 tree-*
       ↓
tree-render.ts          ← 可以导入任何 canvas-*
```

### 1.4 动画与 GSAP 治理

```
▎ 所有 GSAP 调用必须通过 animation-registry.ts
▎ 禁止直接 import gsap（构建时 check-anim.mjs 扫描白名单）
▎ tree-render 内的动画加到 ts = anim.scope('tree-render')
▎ char-rain 使用独立 timeline（anim.timeline()），ts.clear() 不影响
▎ card-stack / orb 的 GSAP 调用走 anim 工具方法
```

## 二、全系统架构模式

### 注册中心与优先级调度

```
所有用户交互通过注册中心分发，禁止绕行：
  GestureRegistry（触摸/鼠标）→ 按 priority 匹配处理器
  KFMState（状态变更）→ 发布-订阅模式
  anim（GSAP 动画）→ 统一隔离层

手势优先级：orb(100) > card-stack(90/80) > page-swipe(50)
```

### 状态机模式（通用）

```
tree-render:        idle ⇄ animating (L.beginOp/endOp)
card-stack:         closed ⇄ opening ⇄ open ⇄ closing（GSAP timeline reverse）
floating-card:      compact → expanding → active ⇄ editing（长按切换）
orb (main):         collapsed ⇄ expanding/collapsing ⇄ expanded ⇄ editing
```

### 浮卡双态交互

```
发射 → compact 态（120×36，仅 BR 光球）
   ↓ 点击 BR 光球
expanding 动画 → GSAP tween（width/height + 角光球动态创建）
   ↓ onComplete
active 态（155×68，四角光球齐全）
   ↓ 长按 BR 光球（600ms）
editing 态 → 右下角缩放手柄，卡片大小跟随
```

### 文件树展开/折叠

```
rebuildTree() → 终端态主树
  → _ensureMetaFromExpandedState() → 补 _fullHeight / _origYs
  → _setupExpandOverlays() → 建 overlay 节点（不碰主树）
  → ts.to() + ts.eventCallback('onComplete') → 动画
  → cleanupCharRain() + removeAllOverlays() + _resetAnimTimeline()

双树渲染:
  overlay 树通过 renderer.setOverlayRoot() 独立渲染
  主树和 overlay 树互不影响
```

## 三、全系统补丁模式（不要这样做）

### ❌ 给 overlay 加 `overflow` 属性

```
问题: 字符雨被容器裁剪
补丁: 在 overlay 上加 overflow: visible
根解: 字符雨建在独立 charLayer 上，不在容器内，不受 overflow 影响
```

### ❌ 拖拽残留状态用 `if` 守卫绕过

```
问题: touchend 丢失（多指/中断）→ _dragItem 残留 → 下次触摸行为异常
补丁: _startFloatingDrag 开头加 if (_dragItem) { clear; exit; null }
根解: 拖拽生命周期应由事件系统保证，
      touchend/mouseup 集中处理 + 错误路径兜底
```

### ❌ 位置与尺寸循环依赖

```
问题: 浮卡弹性边界中 cardLeft = f(newW) 且 newW = f(cardLeft)
补丁: 用上一帧的 _dragItem.cardWidth 做右边界钳制
根解: TL 由 orb - 自然尺寸独立计算，尺寸由 orb - TL 直接得出，
      两者无循环依赖（当前代码已根解）
```

### ❌ 编辑模式面板位置偏移

```
问题: 主光球编辑模式下缩到最小 → 继续拖动 → 面板位置偏移
补丁: 无（旧代码直接用 updatePanelPosition 重算 panelLeft）
根解: 编辑模式下 panelLeft = dragStartPanelX（固定），
      只改变 panelWidth/Height，不改变位置
### ❌ 硬编码常量（路径/名称/数值）

```
问题: 根路径 ' /root ' 分散在 main.ts / tree-model.ts / 测试 / 服务端共 5 处
补丁: 下次改路径时再同步改 5 个地方
根解: 定义成共享常量，一处改全动
```

额外伤害: 硬编码的路径/名称/数值会隐式地成为"契约"。其他模块依赖这个值时，它
          就成了"刚好碰巧一样的巧合"，而不是"本就应该一样的定义"。下次有人改了
          一个地方但漏了另一个，bug 的表现往往是静默的（树渲染为空、API 返回 404），
          很难排查。

判断标准: 同一字面量在 2 个及以上文件中出现 → 必须提取为共享常量。
          同一字面量在 1 个文件中出现 2+ 次 → 也建议提取，至少 const 本地化。

**判断标准**：如果你为了修一个 bug 而新增的代码行数 > 你能用一句话说清的逻辑，那你很可能在打补丁。

## 四、修改代码前的自查清单

在 **任何 .ts 文件** 加任何逻辑前，逐条确认：

- [ ] **这是补丁还是重构？**
  - 补丁：用 `if` 绕过某个边缘情况，不知道根本原因，未来同类 bug 还会出现
  - 重构：找到了根因，通过调整数据模型或生命周期消除它，同类 bug 不可能再出现
  - 判断标准：如果你为了修一个 bug 新增的代码行数 > 你能用一句话说清的逻辑，那你很可能在打补丁

- [ ] **这个改动对未来可维护性有什么影响？**
  - 新增的代码是封装了一个概念，还是分散到多个地方了？
  - 如果一个函数/模块需要知道另一个模块的内部状态才能工作 → 耦合了，需要重新设计
  - 如果下次加同类功能需要复制粘贴改参数 → 说明抽象不到位
  - "代码变少"是正向指标，但不是目标。目标是可维护、可扩展、工程化、系统化

- [ ] **生命周期的完整性**
  - 创建的东西在哪里销毁？
  - 订阅的东西在哪里取消？
  - 如果展开/折叠/打开/关闭可以反复触发，每次触发时之前的状态是否已清理干净？

- [ ] **硬编码常量检查**（路径、名称、数值、Magic Number）：
  - 这个值在代码里出现了几次？在几个文件里？
  - 如果未来需要改它，你能保证找到所有地方吗？
  - 如果在 2 个及以上文件中出现 → 必须提取为共享常量
  - "刚好碰巧一样"不是契约——显式引用的才算
- [ ] **设计验证**（UI/交互改动必问）：
  - 在写任何代码之前，能不能用一句话说清楚最终产物的视觉/交互形态？
  - 这句话你确认过了吗？（我说，你点头或否定，然后我才开始写代码）
  - 如果改动涉及布局——新元素放在哪里？跟谁相邻？跟谁对齐？尺寸如何？

- [ ] **动画安全**（展开/折叠/过渡相关改动必问）：
  - 有 `overlay` 的操作——`_activeOverlays` 的正确性维持了吗？
  - 如果改动在 `onComplete`/`ts.call` 回调中——反向播放时这个回调会误触发吗？
  - 动画锁（`L.isAnimating`）守卫是否覆盖了所有并发路径？

- [ ] **初始化路径检查**（数据模型重构必问）：
  - 你改动的这个值/函数，在页面加载的初始化阶段会被调用吗？
  - 如果会——它的旧版本在那个路径上有默认值保证不崩溃吗？
  - 新版本在那个路径上也能保证不崩溃吗？

- [ ] **功能系统关联检查**（视觉/交互系统重构必问）：
  - 你改动的这个视觉/功能元素，在系统里还有其他变体或表现吗？
  - 它们是否需要保持视觉一致？是否需要保持交互一致？
  - 你确定当前改动覆盖了所有变体吗？

  **常见关联举例：**
  - 卡片堆卡片 ↔ 浮卡（同一视觉原型的不同状态）
  - 光球（AI 聊天）↔ 光球（调试面板）（相同的交互模式）
  - 主树 overlay ↔ 子容器 overlay（相同的动画机制）
  - 侧栏触摸盒子 ↔ canvas 点击（相同的光标触发逻辑）

## 五、新功能开发 SOP

这个 SOP 的目的是"讨论完→确认→再写代码"，而不是"讨论完直接写→写完发现不对"。
关键节点是第 3 步——我输出的设计描述必须经过你确认，我才能开始写代码。
如果第 3 步你发现不对，纠正后回到第 2 步重新出设计。

### 步骤 1 — 需求提出
你：描述你想要的功能。越自由越好。
我：只澄清模糊描述，不问技术细节。

### 步骤 2 — 现状分析
我：读取相关代码，输出受影响文件列表 + 改动点 + 接口边界。
不输出代码，只出结构和位置。

### 步骤 3 — 设计确认（最关键）
我：用一句话描述最终产物的视觉/交互形态。
  - UI：新元素位置/相邻/对齐/尺寸
  - 交互：操作路径/手势触发
  - 生命周期：创建/销毁/打开/关闭时机
你：确认或纠正。确认 → 步骤 4。纠正 → 回到步骤 2。

### 步骤 4 — 改动计划
我：输出精确改动清单（含不做的事），精确到行号范围。
你：确认无遗漏，说"开始"。

### 步骤 5 — 实施
我：按清单实现，每项标记。不做清单外的优化。

### 步骤 6 — 验证
我：构建 + 测试 + 重启。
你：确认功能符合。不符合 → 回到步骤 1 或 3。

## 六、关键文件职责

```
tree-render.ts         — 文件树展开/折叠动画（~1211 行）
card-stack.ts          — 堆叠卡片 + 浮卡系统（~1000 行）
orb.ts                 — 主光球 + AI 对话面板 + 编辑缩放
renderer.ts            — Canvas 渲染引擎（Box 树 → Canvas 2D）
animation-registry.ts  — GSAP 隔离层
gesture-registry.ts    — 触摸事件注册中心
state.ts               — KFMState 发布-订阅
char-rain.ts           — 字符雨粒子动画
click-queue.ts         — 点击事件队列
debug-assert.ts        — 运行时断言
```
