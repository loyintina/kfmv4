# Bug 修复原则 & AI 协作守则

> 通用版 — 任何项目可直接复制使用。

---

## 核心原则

**禁止打补丁。排查到最深层根因，通过调整架构间接消除 bug。**

这个原则来自痛苦的实践经验：对一个不熟悉的项目，AI 辅助修 bug 容易陷入"打补丁→引入新 bug→再打补丁"的死循环。唯一的出路是停下来，理解整个系统的设计意图，然后问自己：**这个类别的 bug 以后还会出现吗？如果会，架构上缺了什么？**

### 如何判断是否在打补丁

| 是补丁 | 是根本解 |
|--------|---------|
| 加个 `if` 判断绕过某个边缘情况 | 发现为什么这个边缘情况会发生，消除根因 |
| 加 `setTimeout` 等在时序问题上 | 发现时序为什么错了，让时序变得确定 |
| 在调用方加一层修正（"这里传的参数不对，我在外面转一下"） | 发现被调用方为什么接收的参数不匹配 |
| 改完代码行数变多 | 改完代码行数变少（或不变） |

> 并不是所有补丁都是坏的。有时补丁是必要的短期措施。**但要清楚自己在打补丁，并且有明确的计划后续做根本修复。**

## AI 协作守则

来自真实项目协作的经验总结。下述 7 条守则适用于任何 AI 辅助开发的场景。

### 1. 不要我以为，要你以为

AI 容易用自己的理解替换设计意图。**准确理解设计者的意图，不要用自己的框架去套。** 宁可先问清楚再行动。

实践中：
- 不要假设"用户肯定想要这个效果"，先确认
- 如果设计者的描述和你的理解有偏差，用具体例子对齐
- 把理解到的设计意图用自己的话复述一遍，等确认

### 2. 要诚实，不要讨好

逐条排查自己可能的偷懒倾向，诚实说出来。

实践中：
- "这个我做过简单版本的考虑，但你的设计要求我做更难但更正确的选择"
- "这个方案我担心某方面可能会出问题"
- 不要说"完全没问题"然后心里有保留

### 3. 要系统性思考，不要局部修补

每次改动前先想清全貌。

实践中：
- 跟哪些文件有关联？
- 会不会引发新的问题？
- 是打补丁还是根本重构？
- 不是"在一个文件里凑合能工作"，而是整个架构逻辑自洽

### 4. 设计要有对称性和完整性

如果系统有"正向"操作，它的"逆向"操作应该是对称的。

实践中：
- 展开和折叠、打开和关闭、进入和退出
- 动画曲线、延迟时间、方向——正反操作应互为镜像
- "怎么开始的，就怎么结束"

### 5. 状态机思维

清晰的状态机是好架构的核心特征。

实践中：
- 明确的状态定义（idle / animating 等）
- 状态变更必须先于视觉变更
- 任何时候中断当前操作，都应能安全回到某个有效状态
- 避免"跳过状态变更直接操控视觉层"

### 6. 白话沟通，不要绕

如果说不清楚某件事，说明自己也没想清楚。

实践中：
- 用口语讲清本质，而非堆砌术语
- 如果对方问了一个问题而你答不上来，直接说"我没想清楚，让我想想"
- 画图比写字好，举例比抽象好

### 7. Bug 修复原则（延伸）

面对 bug 时不要打补丁破坏未来可维护性。

实践中：
- 读取代码找到更深层的原因
- 设计不留技术债的重构方案
- 保证不仅仅是当前的 bug 被修，而是**这一类 bug 都不会再出现**

### 8. 改共享函数前，先列出所有调用方

一个函数被多个地方调用，每个调用方的语境可能不同。不要假设"所有调用方想要的东西是一样的"。

实践中：
- 改之前：追查这个函数的所有调用点，一一列出来
- 逐一确认：这个改动在每个调用点的语境下是否成立
- 如果发现调用方语境不同（比如兄弟和行），考虑拆成不同函数而非加一个参数区分

### 9. 开工前用白话确认职责边界

"这个模块/函数负责什么、不负责什么"这类边界，设计者觉得"不言而喻"，AI 可能完全不知道。

实践中：
- 写一句白话描述"这个模块/函数的职责边界是什么"
- 发给对方确认："我的理解是：overlay 行只负责背景矩形，文字和图标的动画由字符雨统一负责，对吗？"
- 不要边写代码边猜"这个算谁管的"

## 如何排查 Bug 根因

### 第一步：重现

在动手改任何代码之前，先能稳定复现 bug。没有复现手段的修复是不可验证的。

### 第二步：追调用链

从用户操作的入口（点击、触摸、网络响应）开始，逐层追踪到最终执行路径。问自己：

- 数据流向是什么？哪个环节的数据和预期不一致？
- 状态变更的时机正确吗？
- 多个异步操作之间有没有时序依赖？

### 第三步：提取根因类别

找到根因后，给它归类。同类问题应该一次解决：

| 类别 | 典型根因 | 架构级解法 |
|------|---------|-----------|
| 时序竞争 | 异步操作 A 未完成就触发了 B | 状态机 + 事件队列 |
| 隐式契约 | 模块 A 偷偷读写模块 B 的私有属性 | 显式接口 + 类型化 |
| 状态不同步 | 视觉层和状态层的值不一致 | 单一数据源 |
| 资源泄漏 | 动画/定时器未清理 | 统一生命周期管理 |

### 第四步：验证是否真的修了

- 改动后能稳定通过重现步骤
- 想一想"这类 bug 以后还会出现吗？"
- 如果还会出现，说明没修到根因

## 关键不变量（模板）

以下是不变量检查清单，可根据项目实际情况填写：

- [ ] 动画/效果开始前，所有临时状态已清理干净
- [ ] 动画/效果结束后，所有临时状态已清理干净
- [ ] 状态变更函数不会被正在进行的动画打断（或能安全处理）
- [ ] 模块间不通过 `any` / 运行时属性传递数据
- [ ] 异步操作有超时或取消机制
