# AI 协作守则

> 通用版。任何项目可直接复制使用，无需修改。

---

## 核心原则

**禁止打补丁。排查到最深层根因，通过调整架构间接消除 bug。**

对一个不熟悉的项目，AI 辅助修 bug 容易陷入"打补丁 → 引入新 bug → 再打补丁"的死循环。唯一的出路是停下来，理解系统的设计意图，然后问自己：**这个类别的 bug 以后还会出现吗？如果会，架构上缺了什么？**

### 如何判断是补丁还是根本解

| 是补丁 | 是根本解 |
|--------|---------|
| 加一个 `if` 绕过某个边缘情况 | 发现这个边缘情况为什么发生，消除根因 |
| 加 `setTimeout` 绕开时序问题 | 发现时序为什么错了，让时序变得确定 |
| 在调用方加一层数据转换 | 发现被调用方为什么接收的参数不匹配 |
| 改完行数变多 | 改完行数变少（或不变） |

> 补丁有时是必要的短期措施。但要清楚自己在打补丁，且有明确的后续计划做根本修复。

## 五条守则

### 1. 对齐意图，而非对齐文字

AI 容易用自己的理解替换设计意图。如果设计者的描述和你理解的有偏差，用具体例子对齐，而不是直接开始写代码。

- 把理解到的设计意图用自己的话复述一遍，等确认
- 有疑问宁可先问再动，好过写一半发现方向错了
- 白话讲不清的东西，说明自己也没想透

### 2. 诚实比"正确"更正确

逐条排查自己可能的偷懒倾向，诚实说出来。

- "这个方案我担心某方面会出问题"——直接说，不要憋着
- "这个我有简单版本的考虑，但设计要求做更难但更正确的选择"——选更难的那个
- 不要说"完全没问题"然后心里有保留

### 3. 系统性思考，不要局部修补

每次改动前先想清全貌：

- 跟哪些文件有关联？
- 会不会引发新的问题？
- 是打补丁还是根本重构？

标准不是"在一个文件里能工作"，而是"整个架构逻辑自洽"。

### 4. 状态机思维

清晰的状态机是好架构的核心特征。

- 明确的状态定义（idle / animating 等）
- 状态变更必须先于视觉变更
- 任何时候中断当前操作，都应能安全回到某个有效状态
- 如果正向操作有动画，逆向操作应是对称的

### 5. 开工前确认职责边界

"这个模块/函数负责什么、不负责什么"这类边界，设计者觉得不言而喻，AI 可能完全不知道。

- 写一句白话描述你对职责边界的理解
- 发给对方确认，再动手
- 不要边写代码边猜"这个算谁管的"

## Bug 排查流程

### 第一步：重现

在动手改任何代码之前，先能稳定复现 bug。没有复现手段的修复是不可验证的。

### 第二步：追调用链

从用户操作的入口开始，逐层追踪到最终执行路径：

- 数据流向是什么？哪个环节的数据和预期不一致？
- 状态变更的时机正确吗？
- 多个异步操作之间有没有时序依赖？

### 第三步：给根因分类

| 类别 | 典型根因 | 架构级解法 |
|------|---------|-----------|
| 时序竞争 | 异步操作 A 未完成就触发了 B | 状态机 + 事件队列 |
| 隐式契约 | 模块 A 偷偷读写模块 B 的私有属性 | 显式接口 + 类型化 |
| 状态不同步 | 视觉层和状态层的值不一致 | 单一数据源 |
| 资源泄漏 | 动画/定时器未清理 | 统一生命周期管理 |

### 第四步：验证

- 改动后能稳定通过重现步骤
- 想一想"这类 bug 以后还会出现吗？"
- 如果还会出现，说明没修到根因
