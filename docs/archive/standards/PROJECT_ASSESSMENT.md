# KFM v4 — 项目把控理解与建议

> 本文档记录我对 kfmv4 当前状态的理解、对项目方向的把控判断，以及给后续 agent（和项目 owner）的建议。
> 写于完成 UI Element Registry 三层模型填充 + state 动态化 + ai-tools.ts 创建之后。

---

## 一、我对项目的理解

### 1.1 KFM 是什么

KFM v4 是一个**AI 全量生成**的个人工作台原型。它不是文件管理器，不是 AI 聊天客户端，而是一个**探索"AI 能不能替人写一个完整应用"的实验样本**。它的存在意义不是"好用"，而是"看 AI 能走多远"。

这个定位决定了几个重要的性格：
- **架构质量比功能完成度重要**——因为是实验，所以代码结构、文档、约束体系比"能用"更重要
- **文档就是产品**——VISION_AND_ROADMAP、KFM_V4_INVARIANTS、HANDBOOK、UI_ELEMENT_REGISTRY_SPEC 这四篇文档加起来几乎跟代码一样重要，它们是 AI 之间、AI 与人之间传递设计共识的媒介
- **容错率高**——7 张卡片只有 2 张有内容无所谓，3s 动画锁超时暂时留着也没关系

### 1.2 当前处于什么阶段

从 VISION_AND_ROADMAP 的路线图来看：

```
□ 四层模型中的第①层（卡片工作台 UI）→ 约 60% 完成
  已有：Canvas 引擎、手势系统、动画系统、文件树、光球、卡片堆骨架
  缺：完整的卡片插件系统、工具坞、全局输入栏的命令模式
□ 第②层（Agent 适配层）→ 约 5% 完成
  已有：ai-tools.ts 骨架
  缺：WebSocket 通信通道、实际的 function calling 绑定、多 Agent 编排
□ 第③层（Agent 引擎）→ 0%
□ 第④层（KFM 服务端）→ 约 70% 完成（文件 API 基本够用）
```

项目处于**第①层基本可用、第②层刚刚开始搭脚手架**的阶段。UI Element Registry 的完成是第②层的一个关键基础设施——没有它，AI 不知道页面上有什么。

### 1.3 跟 kfmv2/kfmv3 的关系

从旁边的 kfmv2 和 kfmv3 目录看，这是一个经历了多次完整重写的项目。每次重写不是修修补补，而是**带着前一个版本的教训重新开始**。kfmv4 的架构约束（依赖方向、注册中心、动画隔离）是在前三个版本的废墟上长出来的工程智慧。

## 二、我对项目方向的判断

### 2.1 当前最大的瓶颈是什么

不是功能不够多，是**通信通道没打通**。具体来说：

Registry 已经能回答"页面上有什么"，但这个答案在浏览器内存里。AI agent 在服务端拿不到。为了让 AI 真正用上 Registry，需要一个服务端↔浏览器端的实时通信通道。这是整个第②层（Agent 适配层）的第一步，也是卡住后续所有工作的堵点。

### 2.2 什么东西应该优先做

我的优先级排序：

```
P0  打通通信通道（WebSocket 或轮询）——Registry 的数据对 AI 可见
P1  完成内容层的动态化——让 snapshot 中的 content 实时反映页面状态
P2  绑定能力层的实际执行逻辑——让注册的能力可调用
P3  拆分 card-stack.ts（1364行）——降低单文件复杂度
P4  完善卡片系统（填充 card03-card07）——提升功能完成度
```

### 2.3 什么东西不应该做（至少现在不要）

- **不要加 bounds**——除非明确需要"AI 拖拽卡片"的场景，否则 bounds 的维护成本远超收益
- **不要回到"大管家"模式**——v0.1 的 action/epoch/updateState 被砍掉是有充分理由的
- **不要过早优化性能**——项目还在原型阶段，74 个测试都跑不过的优化没有意义
- **不要重写引擎**——v2 Canvas 引擎够用，重写引擎是上一次架构推倒的原因之一

## 三、给后续 agent 的具体建议

### 3.1 如果是接 Registry 的工作

1. **先读 `docs/UI_ELEMENT_REGISTRY_SPEC.md` §S（已实现部分）+ §5（开放问题）**
2. **再读 `docs/notes/REGISTRY_NEXT_AGENT_DISCUSSION.md`**（本轮讨论笔记）
3. **攻 §5.5 通信通道**——WebSocket 是最干净的方案，但需要项目 owner 确认
4. **如果 WebSocket 暂不可行**——轮询方案也可以先跑起来：浏览器每 500ms 把 snapshot POST 到 `/api/ui/snapshot/push`

### 3.2 如果是接其他工作

- **做卡片系统**：先读 `docs/archive/design/VISION_AND_ROADMAP.md` §6（卡片插件规范），现有的 card03-card07 等待填充
- **做动画优化**：先读 `HANDBOOK.md` §4（调试），注意 3000ms 超时这个已知设计缺陷
- **写测试**：先读 `HANDBOOK.md` §5，注意 GSAP mock 失真问题

### 3.3 通用原则

- **文档先行**：改代码前确认对应的文档也同步更新——这个项目里文档跟代码一样重要
- **心法 12 是铁律**：发现补丁就立即根除，不留给"以后"
- **心法 5 是指南针**：改完代码变多了，大概率方向错了
- **先确认再动手**：遇到 §5 级别的开放问题，先写讨论笔记，不要闷头实现

## 四、一句话总结我的感受

> kfmv4 是我见过的最认真、最讲究的 AI 生成项目之一。它的架构纪律和文档质量已经超过了大量人类写的 side project。它现在最需要的不是更多功能，而是**把最后一公里走完**——让 AI 真的能通过 Registry 感知页面、调用能力。打通通信通道是那最后的一公里。
