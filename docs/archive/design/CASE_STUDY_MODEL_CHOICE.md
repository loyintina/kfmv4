# 案例研究：模型选择错误 — 液体粒子 portal 系统绕弯全过程

## 背景

文件树光标边框需要玻璃管液体粒子效果。目标参数：

| 管道 | w | len | speed |
|------|---|-----|-------|
| 上线 | 1 | 12px | 1× |
| 竖线 | 3 | 4px  | 1/2.5× |
| 下线 | 1 | 12px | 1× |

三段独立运行，碰壁消失、另一端同步冒出。

## 时间线

### 阶段 1 — 弧段死胡同（5 提交）

12px 直线胶囊在 R=4 圆角上无法布置。arc stroke / ring fill / concentric glow 全失败。
**错误选择**：修弧段渲染，而不是换模型。

→ 用户提出 **portal 方案**：放弃弧段，三直线。

### 阶段 2 — 错误模型扎根（10 提交）

portal 实施时，**错误假设**："8 粒子在连续环上，跨拐角需要拆分。"

```
overlap 拆分 → path/physical 双坐标系 → cap 居中 → cap 锚点 →
distTop/distBot 二分 → margin 缓冲区 → os-vs-s 修正
```

每一层都是"再修一个 corner case"。用户反复纠正：
- "为什么要切两段"
- "中心是做什么的"
- "你怎么就理解不了"

### 阶段 3 — 正确模型（1 提交）

**正确假设**：三段独立管道，互不感知。粒子中心在哪段就画哪段的形状，
`len = min(segLen, 2 × min(距入口, 距出口))` 自然 clip 边界。

```typescript
if (physC < topW)                  → 上线: w=1, len∈[0,12]
else if (physC < topW + realVert)   → 竖线: w=3, len∈[0,4]
else                               → 下线: w=1, len∈[0,12]
```

**8 个粒子，各产 1 个 LiquidPoint。** 无拆分、无 cap、无锚点、无 margin。

## 根因

| | 错误模型 | 正确模型 |
|---|---|---|
| 粒子 | 连续环上 12px 身体需拆分 | 独立管道，碰壁触发出入 |
| 跨管 | overlap 分裂成 2~3 段 | 不跨管——中心在哪管就画哪管 |
| 拐角 | clamp → 截断 → 拆分 + cap 补丁 | clip 自然缩到 0 |

**两个模型不可调和。** 在用户模型上贴实现者的模型 → 15 提交补丁链 → 最后 1 提交完全重写。

## 代码量

| | 提交数 | 函数 | 行数 |
|---|---|---|---|
| 补丁链峰值 | 15 | `_rangeToPhysicalPoints` 35 行含 cap/锚点/margin | +37 |
| 正确实现  | 1   | 三路 `if/else`                     | -22 |

## 教训

### 心法 8（选自然满足所有约束的方案，不是改动最小的）

portal 最初"只改一行"（5 段弧 → 3 直线），看起来是最小改动。但后续 15 提交都在对抗同一个错误前提（连续环 + 拆分）。
**如果第一次听到"portal"时就问"要不要换模型"而不是"怎么在代码上改"，就不会绕圈。**

### 判断标准

当一个方案需要"再加一层"超过 3 次时，**不是补丁的问题，是模型的问题。**
- 弧段 → arc stroke (1) → ring fill (2) → concentric (3) → 模型错了，不是弧不够大
- cap → anchor (1) → margin (2) → os-vs-s (3) → 模型错了，不是条件不够准

### 用户的纠正 = 模型信号

"为什么要切两段" "中心是做什么的" "你怎么就理解不了"
——这不是细节分歧，是**底层模型冲突**的信号。听见第一声就该停下来检查前提。
